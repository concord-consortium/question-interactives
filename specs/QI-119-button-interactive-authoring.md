# Button Interactive Authoring

**Jira**: https://concord-consortium.atlassian.net/browse/QI-119

**Status**: **Closed**

## Overview

Implement the authoring form for the button interactive, replacing scaffold placeholders with three real fields: Prompt (rich text), Button Label (30-character advisory limit with validation), and Script URL (with helper text). The runtime component is temporarily replaced with a JSON debug view to verify authored state until the runtime story is implemented.

## Requirements

- Replace `exampleAuthoredState` field in the authoring schema with button-specific fields: `buttonLabel` and `scriptUrl`
- **Prompt**: Optional rich text field using the existing `richtext` widget, with help text: `Example: "Click this button when you have finished answering all the questions."`
- **Button Label**: Text input with a 30-character limit (`maxLength: 30`), title "Button Label (limit: 30 characters)", placeholder: `Example: "I'm Done!"`. RJSF will show a validation error if the label exceeds 30 characters; input is not blocked.
- **Script URL**: Textarea (6 rows) with `format: "uri"` validation, placeholder: `Example: "https://example.com/action-script"`, help text below field: "Paste the URL to the action script for this button to run when clicked. Scripts are generated by a developer." Include a description note: "(these parameters may override the above settings)"
- Update `IAuthoredState` interface in `types.ts` to replace `exampleAuthoredState` with `buttonLabel` (optional string) and `scriptUrl` (optional string)
- Update `DefaultAuthoredState` and `DemoAuthoredState` in `types.ts` with appropriate default/demo values
- Remove the `required` field from the authoring schema (buttons don't have submit/lock behavior)
- Remove the `hint` field from the authoring schema (buttons don't have hints)
- Remove `exampleInteractiveState` from `IInteractiveState` in `types.ts` and remove the timer logic from `button.tsx`
- Update the `isAnswered` function if needed based on the removal of submit behavior
- Field ordering in the form: Prompt, Button Label, Script URL
- **Runtime component**: Replace the placeholder ButtonComponent with a temporary debug view that renders a "Temporary Debug View" label followed by `JSON.stringify(authoredState, null, 2)` in a `<pre>` with `white-space: pre-wrap` and monospace font. The label makes it clear to stakeholders that this is a placeholder until the runtime story is implemented.
- The runtime preview in the "Authoring" tab should update as the author edits fields (this is automatic with BaseQuestionApp)
- Verify the demo page (`npm start`) renders correctly with updated authored state values
- All existing tests must pass; update tests to reflect the new authored state fields

## Decisions

### Clean up exampleInteractiveState from IInteractiveState
**Context**: The scaffold included `exampleInteractiveState` in `IInteractiveState` and timer logic in `button.tsx` that were not needed for the button interactive.
**Decision**: Added requirement to remove `exampleInteractiveState` from `IInteractiveState` and the timer logic from `button.tsx`.

---

### required and hint removal scope
**Context**: The spec says to remove `required` and `hint` from the authoring schema. Need to clarify what "remove" means for inherited vs. directly-defined fields.
**Decision**: `required` is inherited from `IAuthoringMetadataBase` and won't be removed from the TS interface — only from the JSON schema so it doesn't appear in the form. `hint` is defined directly on `IAuthoredState` and should be removed from both the interface and schema.

---

### Report item file changes
**Context**: Whether `get-report-item-html.tsx` needs changes for the new authored state shape.
**Decision**: `get-report-item-html.tsx` doesn't reference `exampleAuthoredState` directly — it's a TODO placeholder. No changes needed for this story.

---

### 30-character limit UI behavior
**Context**: Need to specify whether the 30-character limit on Button Label blocks input or shows a validation error.
**Decision**: `maxLength: 30` shows a validation error via RJSF but doesn't block input. This matches RJSF's default behavior.

---

### Demo page verification
**Context**: Need to ensure the demo page renders correctly after all the authored state changes.
**Decision**: Added requirement to verify demo page (`npm start`) renders correctly with updated authored state values.

---

### No accessibility concerns
**Context**: WCAG review of the authoring form and runtime debug view.
**Decision**: The authoring form uses standard RJSF form controls with proper labels via JSON Schema `title` properties. The temporary debug runtime view has no interactive elements. No accessibility issues identified.
